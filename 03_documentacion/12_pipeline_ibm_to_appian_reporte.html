Fase 1 — Ingeniería Inversa del Formato Appian Real

No existía documentación completa del formato interno del ZIP de Appian. Se realizó ingeniería inversa sobre un paquete real exportado del proyecto Profuturo.

📁

Paquete Appian real analizado

Se analizó el archivo NCI Redención BonoSP2.zip (exportación real de Appian v26.3.165.0 del proyecto Profuturo, migración ya concluida).

25 constants · 13 integraciones · 10 rules · 9 interfaces · 27 procesos · 5 record types
🏗️

Descubrimiento clave — Estructura real

La documentación oficial decía appian-package.xml. La realidad fue completamente distinta: carpetas por tipo de objeto con archivos "haul" por UUID.

META-INF/ · application/ · content/ · processModel/ · recordType/ · group/
🔑

UUIDs — Sistema de identidad

Appian identifica objetos por UUID. Si el UUID del paquete coincide con uno existente en el ambiente, actualiza el objeto en lugar de crear uno nuevo.

Se implementó UUID v5 determinístico → misma entrada = mismo UUID siempre
📋

MANIFEST.MF — 3 líneas solamente

La documentación indicaba un manifest complejo. En la práctica son exactamente 3 líneas: versión, versión Appian y fecha de creación.

Manifest-Version: 1.0 · Appian-Version: 26.3.165.0 · Created-On: timestamp
📦

Record Types → RDBMS_TABLE

Los Business Objects de IBM mapean a Record Types en Appian con fuente RDBMS_TABLE. Cada campo IBM se convierte en columna de tabla con tipo Appian.

Datasource: jdbc/Appian · Tabla: UPPER_SNAKE_CASE · Campo: camelCase
⚙️

Constants desde Env Vars

Las variables de entorno de IBM (URLs, IDs, parámetros de config) se convierten en Constants de Appian. Las que contienen URL/HOST se marcan como environmentSpecific.

71 variables IBM → 71 constants Appian en carpeta organizada

Fase 2 — Análisis del Formato IBM TWX

El archivo .twx es también un ZIP con XMLs. Se descubrió que las herramientas CLI existentes no extraían los campos de los BOs. Se construyó un parser directo.

🔢

Prefijos de tipo en IBM

Cada artefacto en el TWX se nombra con un prefijo numérico que indica su tipo:

12.* = Business Objects  |  25.* = BPDs  |  1.* = Services  |  64.* = Coach Views  |  62.* = Env Vars
🧬

Estructura XML de un BO IBM

Los campos del BO están en <teamworks><twClass><definition><property>. El tipo está en <annotation><typeName>, no como atributo directo.

El CLI ibm_twx_tools devolvía fields:[] vacíos → se construyó IBMTwxParser propio
🌿

Variables de entorno IBM

Están en archivos 62.*.xml bajo el tag <environmentVariableSet><envVar> con <envVarDefault><value>. El proyecto tiene 71.

URLs de WMB, IDs de subetapas, parámetros de negocio → Constants Appian

Fase 3 — Pipeline de Transformación (5 Pasos)
1

Parser IBM TWX → Extracción de datos

Se abre el .twx como ZIP y se parsean los XMLs internos directamente. Se extraen todos los Business Objects con sus campos tipados y todas las variables de entorno con sus valores por defecto.

Módulo: IBMTwxParser
Input: archivo .twx (ZIP de IBM BPM)
Output: lista de IBMBusinessObject (con campos) + lista de IBMEnvVar
Profuturo: 54 BOs con campos + 71 variables de entorno extraídas
2

Env Vars → Constants de Appian

Cada variable de entorno IBM se convierte en una Constant de Appian. Las variables que contienen URLs, hosts o puertos se marcan automáticamente como environment-specific (configurables por ambiente).

Módulo: ConstantsBuilder
Detección automática ENV: si el nombre contiene URL, HOST, IP, PORT, PATH, KEY, SECRET
XML generado: <contentHaul><constant><typedValue>...</constant></contentHaul>
Profuturo: 71 constants en carpeta NCI_RB Constants
3

Business Objects → Record Types

Cada BO de IBM se convierte en un Record Type de Appian respaldado por tabla RDBMS. Los campos se mapean con sus tipos Appian equivalentes. Los nombres de tabla siguen la convención UPPER_SNAKE_CASE.

Módulo: RecordTypeBuilder
Fuente: RDBMS_TABLE · datasource: jdbc/Appian
Nombre tabla: NCI_RB_NombreBO (UPPER_SNAKE_CASE)
XML generado: <recordTypeHaul><recordType><sourceConfiguration>...</recordTypeHaul>
Profuturo: 54 Record Types generados
4

Services → Expression Rule Stubs

Por cada BO se genera un Expression Rule stub con los inputs tipados correctamente y un comentario TODO indicando qué lógica IBM debe traducirse. El equipo Appian puede partir de estas reglas ya creadas y estructuradas.

Módulo: RuleBuilder
Patrón generado: NCI_RB_get{NombreBO}ById(id_int: Integer)
Definition: stub con comentario TODO referenciando el servicio IBM original
Profuturo: 54 rule stubs en carpeta NCI_RB Rules
5

Ensamblado del ZIP Appian

Se genera el applicationHaul con el inventario completo de UUIDs, se escribe el MANIFEST.MF con la versión de Appian correcta y se ensambla el ZIP con todos los objetos en sus carpetas correspondientes.

Módulos: ManifestBuilder + ZipPackager
UUID strategy: UUID v5 determinístico — mismo TWX = mismo UUID siempre (deploys idempotentes)
Output Profuturo: NCI_RB_appian_package.zip · 183 archivos · 221 KB
Compatible con: Appian v26.3.165.0 (versión del cliente)
